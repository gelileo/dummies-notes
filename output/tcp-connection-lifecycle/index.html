TCP connection lifecycle

The full set of steps two computers go through to talk reliably over a network: they first greet each other to agree to start, then send messages back and forth while confirming each one arrived, and finally say goodbye to close the conversation cleanly.

Communication protocol

An agreed set of rules two computers follow when talking, so each one knows what kind of message to expect next and what every message means.

Data packets

Small labeled chunks that a message is chopped into before being sent across a network, each traveling on its own and carrying enough information to be put back in order.

Already covered — see its figure.

Unreliable delivery

The fact that a message sent across a network can quietly fail to arrive, arrive damaged, or arrive out of order, with no automatic warning to the sender.

Builds on:

  • Computer network (computer-network: not yet covered — left for a later run)
  • Data packets — Messages do not travel as one piece; they travel as many independent chunks, which is the direct reason a delivery can lose some pieces, damage one, or have them arrive out of order.

Delivery acknowledgement

A short reply one computer sends back to confirm it received what the other sent, so the sender knows the message arrived and does not have to guess or resend it blindly.

Builds on:

  • Data packets — A delivery acknowledgement confirms that a specific sent chunk arrived, so the reader first needs to picture a message as discrete chunks that travel separately before a per-chunk 'got it' reply means anything.
  • Unreliable delivery — An acknowledgement exists only to solve this problem; if delivery were always guaranteed the reply would be pointless, so the reader must first understand that sent messages can silently go missing.

TCP connection lifecycle

The full set of steps two computers go through to talk reliably over a network: they first greet each other to agree to start, then send messages back and forth while confirming each one arrived, and finally say goodbye to close the conversation cleanly.

Builds on:

  • Data packets — A TCP conversation does not send a message in one piece; it ships many small chunks and confirms each one arrived, so the reader must first picture what those chunks are before the greet-confirm-goodbye steps make sense.
  • Communication protocol — The whole lifecycle is one particular set of agreed rules, so the reader needs the general idea that machines follow a shared rulebook before seeing the specific greet-confirm-goodbye rules TCP uses.
  • Delivery acknowledgement — TCP is called reliable precisely because every chunk is answered with a confirmation, and if no confirmation comes the chunk is sent again, so the reader cannot understand why the lifecycle is trustworthy without this back-and-forth confirming idea.
